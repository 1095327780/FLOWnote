# FLOWnote 0.5.0 — Provider Abstraction & Self-Contained Agent Runtime

**Status:** Draft · 2026-05-14
**Owner:** @1095327780 (阿浩)
**Target release:** 0.5.0-alpha
**Supersedes:** none

> This document specifies the redesign that lets FLOWnote run **without** any external
> CLI (OpenCode, Claude Code, etc.) — users install the plugin, paste an API key, and
> the chat + skill workflows work immediately, including on iOS/Android. The current
> OpenCode integration is kept as one of multiple providers, not the only option.

---

## 1. Goals

1. **Zero external dependency for the default install path.** A user with Obsidian
   alone should be able to install FLOWnote, paste an API key, and use chat, skills,
   and capture without installing OpenCode, Claude Code, or any other CLI.
2. **First-class mobile support.** Anthropic API calls and all skill workflows must
   run on Obsidian iOS / Android (no Node, no `child_process`, no filesystem
   beyond Obsidian's Vault API).
3. **Preserve OpenCode mode.** Existing users on OpenCode keep working. Provider is
   a setting; default for new installs is the direct API path.
4. **Preserve the 14 `ah-*` skills.** Skill markdown content is the asset. The
   runtime that invokes them changes; the skills do not.
5. **Production-shaped agent loop.** Streaming, tool calls, permission gates,
   conversation persistence — same UX as today, just running locally inside the
   plugin.
6. **Provider-pluggable.** Add new providers (DeepSeek, GLM, Moonshot via
   OpenAI-compat) without touching agent-loop or tool code.

## 2. Non-goals

- **Sub-agents / forked contexts.** Single-agent loop only in 0.5.0. Multi-agent
  orchestration is deferred.
- **MCP server integration via subprocess.** Deferred to a later release.
  HTTP-based MCP (if/when standardized) can be revisited later.
- **Shell / Bash tool.** Not part of the FLOWnote scope. Users who need shell
  power can keep OpenCode mode.
- **Aggressive auto-compaction.** Modern models ship with 200K–1M context;
  context compression is opportunistic, not foundational. v0.5.0 ships with a
  simple "remind user when approaching limit" mechanism and a manual
  `/compact` command. Automatic mid-conversation compaction lands in 0.6.x if
  real usage demands it.
- **OAuth / Claude.ai account login.** Plain API keys only. Multi-account UX
  later if needed.

## 3. Current state (one screen)

```
Obsidian plugin (Electron renderer)
        │
        ├─ FlowNoteClient ──► sdk-transport.js ──► OpenCode HTTP API (localhost:port)
        │                                                  │
        │                                          OpenCode CLI process
        │                                                  │
        │                                          model providers (Anthropic, etc.)
        │

```

Hard dependencies:

- Desktop only (OpenCode CLI must be installed).
- `child_process` / `os.homedir` / env-var sniffing to locate the CLI binary.
- All AI calls leave the plugin → OpenCode → upstream. Plugin is a UI on top.

## 4. Target state

```
Obsidian plugin (works on desktop + iOS + Android)
        │
        ├─ FlowNoteClient
        │     │
        │     ├─ provider-registry ───► AnthropicDirectProvider
        │     │                          OpenAICompatProvider
        │     │                          OpenCodeLegacyProvider  (existing transport)
        │     │
        │     ├─ agent-loop ──► tool-registry ─► tools/* (Vault API only)
        │     │      │
        │     │      └─ skill-runner ──► loads SKILL.md from bundled-skills/,
        │     │                          injects as system addendum, restricts
        │     │                          tool whitelist for that turn
        │     │
        │     ├─ permission-gate ──► UI modal for destructive tools
        │     └─ session-manager ──► persisted to plugin data.json

```

Hard requirements removed: `child_process`, `os.*`, env scanning, external
process management. The only "system" thing the plugin still does is locate the
OpenCode binary — but **only in OpenCode mode**, gated behind that provider.

## 5. Key technical decision: HTTP transport

**All upstream HTTP MUST go through Obsidian's `requestUrl()` (from the
`obsidian` module), not `fetch` and not the official `@anthropic-ai/sdk`.**

Reasons:

1. Obsidian's renderer has CSP rules. `fetch()` to arbitrary hosts is blocked on
   mobile and unreliable on desktop with certain proxy configurations.
2. `requestUrl()` works identically on desktop, iOS, Android.
3. The official SDK is mostly type-safety + a thin HTTP wrapper. Reimplementing
   that surface as a 200-line `anthropic-client.js` is cheaper than fighting CORS
   and shipping `node-fetch` shims.
4. Bundle size: avoiding `@anthropic-ai/sdk` saves ~150 KB minified.

We will copy/adapt only the **type surface** we need (request/response shapes,
streaming event union) — those are derived from the public API docs, not from
the SDK source. No SDK code is imported.

For streaming, Anthropic's API uses Server-Sent Events. `requestUrl()` does not
stream by default. We will use **Obsidian's `request()` low-level helper with
`responseType: 'text'` and `streaming: true`** where available, or fall back to
a custom SSE parser over `Readable` (desktop) and chunked-text accumulation
(mobile). The streaming abstraction is hidden behind the `Provider` interface
so callers see a single async iterator regardless of platform.

## 6. Provider interface

`runtime/providers/provider.js` (interface, JSDoc-typed):

```js
/**
 * @typedef {Object} ProviderMessage
 * @property {'user'|'assistant'|'system'} role
 * @property {ContentBlock[] | string} content
 *
 * ContentBlock is the Anthropic-shaped block union:
 *   { type: 'text', text }
 *   { type: 'tool_use', id, name, input }
 *   { type: 'tool_result', tool_use_id, content, is_error? }
 *   { type: 'image', source: { type, media_type, data } }   // v2
 */

/**
 * @typedef {Object} CreateMessageInput
 * @property {string} model
 * @property {ProviderMessage[]} messages
 * @property {string|ContentBlock[]} system
 * @property {ToolSpec[]} tools
 * @property {number} maxTokens
 * @property {number} [temperature]
 * @property {AbortSignal} [signal]
 */

/**
 * @typedef {Object} StreamEvent
 * @property {'message_start'|'content_block_start'|'content_block_delta'|
 *           'content_block_stop'|'message_delta'|'message_stop'|'error'|'ping'} type
 * @property {Object} [delta]
 * @property {Object} [content_block]
 * @property {Object} [message]
 * @property {Object} [error]
 */

/**
 * Every provider exports:
 *
 *   id          : string   ('anthropic-direct' | 'openai-compat' | 'opencode-legacy')
 *   capabilities: { streaming, tools, vision, jsonMode, ... }
 *   models      : () => Promise<ModelInfo[]>     // list of available models
 *
 *   createMessage(input: CreateMessageInput): AsyncIterable<StreamEvent>
 *     yields normalized Anthropic-shaped events; provider adapters translate
 *     OpenAI-compat chunks etc. into this shape
 *
 *   countTokens(messages): Promise<number>       // best-effort; tiktoken or
 *                                                  provider-side endpoint
 *
 *   testConnection(): Promise<{ ok, error?, latencyMs }>
 *     used by the diagnostics panel
 */
```

The agent loop only ever sees this interface. **Adding a new provider is a new
file in `runtime/providers/` plus a registry entry. Nothing else changes.**

### Provider implementations

| Provider | Module | Notes |
|---|---|---|
| `anthropic-direct` | `runtime/providers/anthropic-direct.js` | Calls `https://api.anthropic.com/v1/messages` via `requestUrl`. Native Anthropic shape — no translation. |
| `openai-compat` | `runtime/providers/openai-compat.js` | Translates Anthropic-shape messages ↔ OpenAI chat-completions shape. Used for DeepSeek / GLM / Moonshot / SiliconFlow / any user-supplied base URL. |
| `opencode-legacy` | `runtime/providers/opencode-legacy.js` | Adapter around existing `runtime/sdk-transport.js`. Wraps existing event stream into the new `StreamEvent` shape. |

Translation table for OpenAI-compat (just the essentials):

| Anthropic concept | OpenAI-compat equivalent |
|---|---|
| `messages[].role = 'user'/'assistant'` | same |
| `content: string` | `content: string` (passthrough) |
| `content: [{type:'text',text}, ...]` | concat texts → string |
| `content: [{type:'tool_use',id,name,input}]` | `tool_calls: [{id, type:'function', function:{name, arguments: JSON.stringify(input)}}]` |
| `content: [{type:'tool_result',tool_use_id,content}]` | `role:'tool', tool_call_id, content` |
| `system: string` | `messages.unshift({role:'system',content})` |
| `tools: [{name, description, input_schema}]` | `tools: [{type:'function',function:{name,description,parameters}}]` |
| stream events `content_block_delta` | `delta.tool_calls[].function.arguments` chunks accumulated then emitted as one block |

Vision (image inputs) is NOT in 0.5.0 scope; the translation table can be
extended in 0.6.x.

## 7. Tool registry & contract

`runtime/agent/tool-registry.js`:

```js
const TOOL_DEFAULTS = {
  isEnabled:        () => true,
  isReadOnly:       () => false,   // fail-closed: assume side-effects
  isDestructive:    () => false,   // explicit opt-in for destructive flag
  isConcurrencySafe:() => false,   // assume not parallelizable
  checkPermissions: async () => ({ behavior: 'allow' }),
  validate:         async () => ({ ok: true }),
  userFacingName:   (input) => '',
};

/**
 * @typedef {Object} ToolDef
 * @property {string} name              kebab-case identifier shown to model
 * @property {string} description       prompt shown to the model
 * @property {Object} inputSchema       JSON Schema for the tool input
 * @property {(input,ctx)=>boolean} [isEnabled]
 * @property {(input)=>boolean}     [isReadOnly]
 * @property {(input)=>boolean}     [isDestructive]
 * @property {(input)=>boolean}     [isConcurrencySafe]
 * @property {(input,ctx)=>Promise<PermissionResult>} [checkPermissions]
 * @property {(input,ctx)=>Promise<ValidationResult>} [validate]
 * @property {(input,ctx)=>AsyncIterable<ProgressOrResult>}   execute
 * @property {(input)=>string}      [userFacingName]
 */

export function buildTool(def) {
  return { ...TOOL_DEFAULTS, ...def };
}

export const toolRegistry = new Map();   // name → Tool
export function registerTool(tool) { toolRegistry.set(tool.name, tool); }
```

`execute` is an **async generator** that yields:

- `{ type: 'progress', message, data? }` — UI hint, optional
- `{ type: 'result', content, isError? }` — terminal result that becomes the
  `tool_result` content block in the next API turn

Pattern lifted directly from Claude Code's `ToolDef`. The defaults are
fail-closed (assume not safe, not read-only, not concurrency-safe) so adding a
new tool that forgets to declare a flag does the safe thing.

### 7.1 Tool catalog (the seven)

| Tool name | Replaces (Claude Code) | Read-only? | Destructive? | Implementation |
|---|---|---|---|---|
| `vault_read` | `FileReadTool` | ✅ | ❌ | `app.vault.cachedRead(file)` |
| `vault_write` | `FileWriteTool` | ❌ | ✅ (if file exists) | `vault.create` or `vault.modify` |
| `vault_edit` | `FileEditTool` | ❌ | ✅ | Read → string replace → `vault.process()` (atomic) |
| `vault_list` | `LSTool` + `GlobTool` | ✅ | ❌ | `vault.getMarkdownFiles()` + glob match + frontmatter filter |
| `vault_search` | `GrepTool` | ✅ | ❌ | Pass 1: `metadataCache` (tags/links/headings). Pass 2: content scan via `vault.cachedRead` on the matched subset |
| `ask_user` | `AskUserQuestionTool` | ✅ | ❌ | Renders an inline `<select>` / `<button>` panel in the chat view. Returns user's selection |
| `skill_invoke` | `SkillTool` | depends on skill | depends | Loads `bundled-skills/<name>/SKILL.md`, runs the skill as a constrained sub-turn (see §8) |

Web access (`web_fetch`) is deferred to 0.6.0 — useful but not core.

Each tool gets its own file under `runtime/agent/tools/`. Each is ~80–150 lines
including JSON Schema + JSDoc + tests.

### 7.2 Permission model

Three behaviors, evaluated by `checkPermissions(input, ctx)`:

- `{ behavior: 'allow' }` — proceed silently
- `{ behavior: 'deny', reason }` — return as `tool_result.is_error = true`
- `{ behavior: 'ask', summary, choices? }` — pause the loop, show a modal/inline
  panel in the chat view, wait for user choice. Choices default to
  `['allow once', 'allow for this conversation', 'deny']`. Persistence of
  per-conversation grants lives on `SessionState.grants`.

Defaults:

- All `isReadOnly` tools → `allow`.
- `vault_write` to a new path → `ask` (first time per conversation).
- `vault_write` / `vault_edit` to an existing path → `ask`.
- `skill_invoke` → `allow` (the skill itself can prompt).

Settings exposes a global `permissionMode`:
`acceptAll` | `acceptReadOnly` (default) | `askAlways`.

## 8. Skill runner

A skill is a directory under `bundled-skills/<name>/` with:

- `SKILL.md` — the prompt
- `references/*` — long-form references the skill prompt links to (read
  on-demand by the model via `vault_read` after the skill prompt directs it)
- `assets/*` — templates used by the skill workflow

`runtime/agent/skill-runner.js` orchestrates `skill_invoke`:

```
on tool_use { name: 'skill_invoke', input: { skill: 'ah-card', args } }:
  1. Resolve skill = loadBundledSkill('ah-card')
     - returns { promptText, referencesRoot, allowedTools, model? }
  2. Build a sub-conversation:
       systemAppendix = promptText + skillContextHeader
       tools          = filterRegistry(allowedTools)     // restricts surface
       initialUser    = `Skill arguments: ${args}`
  3. Run agent-loop with this sub-conversation
       - runs to completion (own turn budget, max 20 turns)
       - shares the same provider + session storage but tagged with skillId
  4. Final assistant text is returned as the tool_result.content
```

The parent loop sees this as one `tool_use` → `tool_result` round-trip, same as
any other tool call. The user sees in the UI: a collapsible "Used skill: ah-card"
panel with the sub-conversation inside.

**No change to bundled-skills source**: every `SKILL.md` you already wrote runs
unchanged. The `allowedTools` field is parsed from the skill's frontmatter (or
defaults to all vault tools if missing).

### 8.1 Skill discovery & sync

Stay with the current model:

- Bundled skills ship inside the plugin (already embedded as `utf8` strings —
  the scorecard fix in 0.4.0).
- On plugin load, sync to `<vault>/.opencode/skills/<name>/` (path may be
  renamed to `.flownote/skills/` in 0.5.0 to drop the OpenCode reference, with
  a one-time migration that copies from the old location).
- Skill loader prefers the vault copy (user-editable) over the bundled fallback.

## 9. Agent loop

`runtime/agent/agent-loop.js`. Pseudocode (~150 lines real):

```js
async function* runAgentLoop({ provider, session, system, userMessage, tools, ctx }) {
  session.append(userMessage);

  for (let turn = 0; turn < ctx.maxTurns; turn++) {
    const stream = provider.createMessage({
      model:    session.model,
      messages: session.toApiMessages(),
      system,
      tools:    tools.map(t => t.toJsonSchemaSpec()),
      maxTokens: session.maxTokensPerTurn,
      signal:    ctx.signal,
    });

    const assistantBlocks = [];
    const toolUses        = [];

    for await (const ev of stream) {
      yield { type: 'stream', event: ev };
      // accumulate text blocks and tool_use blocks
      if (ev.type === 'content_block_stop' && ev.content_block.type === 'tool_use') {
        toolUses.push(ev.content_block);
      }
      // ... text accumulation
    }

    session.appendAssistant(assistantBlocks);

    if (toolUses.length === 0) return;        // model is done

    const toolResults = [];
    const [parallel, serial] = partition(toolUses, u => isReadOnlyToolUse(u));

    // Run read-only ones in parallel, destructive ones serially
    const parallelResults = await Promise.all(parallel.map(u => runTool(u, ctx)));
    for (const u of serial) toolResults.push(await runTool(u, ctx));

    session.appendUser([...parallelResults, ...toolResults]);
  }

  yield { type: 'error', error: new MaxTurnsExceeded() };
}
```

### 9.1 Streaming UX

Each stream event is forwarded to the chat view, which already handles:

- `content_block_delta` of type `text` → append to live message bubble
- `tool_use` block start → render a "🛠 Using `vault_read`…" chip
- `tool_use` block stop → show result preview

Existing `runtime/view/` rendering needs minimal change — only the event shape
adapter.

### 9.2 Cancellation & retry

- `AbortController` propagates to `requestUrl` (Obsidian's request can be
  aborted via signal in newer versions; fallback: stop reading the stream and
  return a synthetic `message_stop`).
- Retry on `5xx` and on network errors: 3 attempts with exponential backoff
  (300ms · 900ms · 2700ms), only for the request to the model API. Tool errors
  are NOT retried — they're returned to the model as `tool_result.is_error`,
  and the model decides what to do.

### 9.3 Context budget

Modern Anthropic models support 200K context; some models advertise 1M.
v0.5.0 does the simple thing:

- Track approximate token count per session (rough heuristic: 1 token ≈ 3.5
  chars CJK, 4 chars Latin).
- At 80% of the model's window, show a non-blocking banner: "Conversation is
  getting long — consider /compact or starting a new session."
- `/compact` is a user-invocable command that asks the model to summarize the
  conversation, replaces older turns with the summary, keeps the last 4 turns
  verbatim.
- No auto-compact in v0.5.0. Revisit only if telemetry-free user feedback says
  it's needed.

## 10. Session storage

Existing `runtime/session-store.js` already persists sessions to disk. Format
change:

```jsonc
// data/sessions/<id>.json
{
  "id": "uuid",
  "createdAt": "...",
  "providerId": "anthropic-direct",
  "model": "claude-sonnet-4-6",
  "skillContext": null,                   // or { name, sub: [...] } for skill turns
  "grants": { "vault_write:*": "session" },
  "messages": [
    { "role": "user",      "content": [{ "type": "text", "text": "..." }], "ts": ... },
    { "role": "assistant", "content": [...], "stop_reason": "end_turn", "ts": ... },
    { "role": "user",      "content": [{ "type": "tool_result", "tool_use_id": "...", "content": "..." }] },
    ...
  ]
}
```

Anthropic-shape is canonical on disk. Loading an OpenAI-compat-only session
costs nothing because we always store the normalized shape; the provider
translates only at egress.

Migration from current OpenCode-shape sessions: a one-time `state-migrations`
entry rewrites the message format on first load of 0.5.0.

## 12. Mobile considerations

| Concern | Resolution |
|---|---|
| No Node APIs (`fs`, `child_process`, `os`) | All vault tools use `app.vault.*`. No node-only deps in the new code path. |
| CORS | All upstream calls via `requestUrl` — bypasses CORS. |
| Streaming | Provider abstracts SSE; on mobile, fall back to non-streaming if `requestUrl` doesn't expose chunks. UX degrades gracefully (response appears all at once). |
| API key storage | Plugin `data.json` — same place desktop uses. |
| Skill files | Already extracted to vault as UTF-8 markdown (0.4.0 fix). Mobile reads them via Vault API like any other markdown. |
| File size | Plugin bundle expected to grow from ~1.5 MB to ~1.7–1.9 MB after adding agent runtime. Within Obsidian's tolerance; well below mobile constraints. |

Gating: `manifest.json` stays `isDesktopOnly: false`. The OpenCode provider
**internally** flags itself as desktop-only and disables itself on mobile;
Anthropic / OpenAI-compat providers work on both.

## 13. Build & bundle impact

Current bundle: 1.5 MB (post-utf8 fix).

Estimated additions:

- `runtime/agent/*` — 1500–2000 lines source → ~30 KB after esbuild
- `runtime/providers/*` — 3 providers, ~600 lines → ~10 KB
- New tool implementations (7 tools) — ~800 lines → ~15 KB
- Net: **+50–60 KB**. Total bundle ~1.55–1.6 MB.

exactly because OpenCode-legacy provider still needs it. No `node_modules`
changes beyond what's already there.

`scripts/guard-file-size.js` thresholds: bump main.js soft limit from 2 MB to
2.5 MB during 0.5.0 development. Revisit before release.

## 14. Risks & mitigations

| Risk | Likelihood | Mitigation |
|---|---|---|
| `requestUrl` SSE streaming behaves differently desktop vs mobile | Medium | Provider abstracts streaming. Mobile fallback returns full response after API completes (no streaming UX), still functional. Test in M1. |
| Tool-call accuracy with smaller models (DeepSeek / GLM) | Medium | Document recommended models. Surface a "tool support unreliable on this model" warning in settings when a known-weak model is picked. |
| Skill sub-loop turn explosion | Low | Hard cap at 20 sub-turns per skill_invoke. Skills that exceed return a soft error. |
| Existing user breaks on upgrade | Low | OpenCode legacy provider stays. Auto-migration sets provider correctly. State-migration rewrites session shape. |
| Bundle size scorecard regression | Low | Already budgeted (+60 KB). |
| Obsidian community policy: external HTTP from plugin | None | All endpoints disclosed in README (already done). Same as Smart Connections, Copilot, Templater. |
| Anthropic API ToS | None | Plain API key usage from a user's account is the documented path. No source from Claude Code is used. |

## 15. Testing strategy

- **Unit tests** under `tests/runtime/agent/` covering:
  - tool-registry defaults are fail-closed
  - each tool's `validate` rejects bad input
  - each tool's `checkPermissions` returns the expected shape
  - provider translators: round-trip a known Anthropic message ↔ OpenAI shape ↔ Anthropic
- **Integration tests** with a `MockProvider` that scripts a fixed event
  stream — covers the agent loop, skill-runner, permission gate.
- **Live smoke test** (manual, gated by env var so CI doesn't burn credits):
  one full round-trip via Anthropic Direct, one via OpenAI-compat (DeepSeek),
  one via OpenCode-legacy.
- **Mobile smoke**: install on iOS + Android, run the same three skills
  (`ah-note`, `ah-card`, `ah-review`) under each provider.

CI gates additions:

- `npm run ci` runs the new unit + mock-integration suites.
- `npm run test:e2e` (new, opt-in): runs the live smoke test if
  `FLOWNOTE_E2E=1` and keys are set. Not on PRs by default.

2. **Default model alias**: keep `sonnet` as default for Anthropic Direct;
   what's the default for OpenAI-compat? Probably `deepseek-chat` because
   it's cheap + works well — confirm with author.
3. **Settings layout**: one provider active at a time, or per-skill provider
   override? Lean: single active provider for v0.5.0, per-skill override in
   v0.6.0 only if real demand.
4. **`requestUrl` streaming on mobile**: needs a 1-day spike in M1 to confirm
   the streaming behavior. If broken on mobile, accept non-streaming UX there
   for v0.5.0; address in v0.5.1.
