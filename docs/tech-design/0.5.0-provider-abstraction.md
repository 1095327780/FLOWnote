# FLOWnote 0.5.0 — Provider Abstraction & Self-Contained Agent Runtime

**Status:** Approved · 2026-05-14
**Owner:** @1095327780 (阿浩)
**Target release:** 0.5.0-alpha

> This document specifies the redesign that lets FLOWnote run **without** any external
> CLI (OpenCode, Claude Code, etc.) — users install the plugin, paste an API key, and
> the chat + skill workflows work immediately, including on iOS/Android. The current
> OpenCode integration is kept as one of multiple providers, not the only option.

## 1. Goals

1. **Zero external dependency for the default install path.** A user with Obsidian
   alone should be able to install FLOWnote, paste an API key, and use chat,
   skills, and capture without installing OpenCode, Claude Code, or any other
   CLI.
2. **First-class mobile support.** Anthropic Messages API calls and all skill
   workflows must run on Obsidian iOS / Android (no Node, no `child_process`,
   no filesystem beyond Obsidian's Vault API).
3. **Preserve OpenCode mode.** Existing users on OpenCode keep working.
   Provider is a setting; default for new installs is the direct API path.
4. **Preserve the 14 `ah-*` skills.** Skill markdown content is the asset.
   The runtime that invokes them changes; the skills do not.
5. **Production-shaped agent loop.** Streaming, tool calls, permission gates,
   conversation persistence — same UX as today, just running locally inside
   the plugin.
6. **Provider-pluggable, with first-class support for Chinese coding plan
   subscriptions.** Built-in presets for Anthropic, OpenAI, DeepSeek,
   智谱 GLM, MiniMax, Moonshot Kimi, 通义千问, 字节豆包, plus a generic
   "custom OpenAI-compat" entry. Each provider with a coding plan offering
   exposes a **two-mode** UX (Coding Plan / Pay-per-token API) so users can
   route their existing subscription through FLOWnote without any client
   impersonation.

## 2. Non-goals

- **Sub-agents / forked contexts.** Single-agent loop only in 0.5.0.
  Multi-agent orchestration is deferred.
- **MCP server integration via subprocess.** Deferred to a later release.
  HTTP-based MCP (if/when standardized) can be revisited later.
- **Shell / Bash tool.** Not part of the FLOWnote scope. Users who need shell
  power can keep OpenCode mode.
- **Aggressive auto-compaction.** Modern models ship with 200K–1M context;
  context compression is opportunistic, not foundational. v0.5.0 ships with a
  simple "remind user when approaching limit" mechanism and a manual
  `/compact` command. Automatic mid-conversation compaction lands in 0.6.x if
  real usage demands it.
- **OAuth / Claude.ai account login.** Plain API keys only. Multi-account UX
  later if needed.
- **Claude Code / OpenCode client impersonation against Anthropic.**
  Anthropic explicitly bans third-party use of Claude Pro / Max subscription
  tokens (Consumer ToS, enforced 2026-01-09 with server-side client
  fingerprinting). FLOWnote will not ship spoofed `User-Agent` headers or
  cookie/OAuth replay against `api.anthropic.com`. Chinese-provider coding
  plans are a different situation — see §6.3.

the OpenCode binary — but **only in OpenCode mode**, gated behind that
provider.

## 5. Key technical decision: HTTP transport

**All upstream HTTP MUST go through Obsidian's `requestUrl()` (from the
`obsidian` module), not `fetch` and not the official `@anthropic-ai/sdk`.**

Reasons:

1. Obsidian's renderer has CSP rules. `fetch()` to arbitrary hosts is blocked
   on mobile and unreliable on desktop with certain proxy configurations.
2. `requestUrl()` works identically on desktop, iOS, Android.
3. The official SDK is mostly type-safety + a thin HTTP wrapper.
   Reimplementing that surface as a 200-line `anthropic-client.js` is
   cheaper than fighting CORS and shipping `node-fetch` shims.
4. Bundle size: avoiding `@anthropic-ai/sdk` saves ~150 KB minified.

We will copy/adapt only the **type surface** we need (request/response
shapes, streaming event union) — those are derived from the public API docs,
not from the SDK source. No SDK code is imported.

For streaming, Anthropic's API uses Server-Sent Events. `requestUrl()` does
not stream by default. We will use **Obsidian's low-level `request()` helper
with chunked-text accumulation** where available, or fall back to single-shot
non-streaming on mobile if necessary (1-day spike in M1 to confirm). The
streaming abstraction is hidden behind the `Provider` interface so callers
see a single async iterator regardless of platform.

## 6. Provider abstraction

### 6.1 Architecture

Every provider is data-driven config + a tiny code path that knows how to
speak one of three protocols:

| Protocol | Used by | Wire format |
|---|---|---|
| `anthropic-messages` | Anthropic native + every Chinese coding plan provider (`/anthropic` endpoints) + custom Anthropic-compat URLs | `POST /v1/messages`, Messages API body, SSE for streaming |
| `openai-chat` | OpenAI native + many Chinese providers' alternate `/v1` endpoint + custom OpenAI-compat URLs | `POST /v1/chat/completions`, Chat Completions body, SSE for streaming |
| `opencode-runtime` | Existing OpenCode integration | Existing transport, wrapped as a `Provider` adapter |

The agent loop only ever speaks **Anthropic Messages shape internally**. The
`openai-chat` provider adapter translates at the egress/ingress boundary; the
`opencode-runtime` adapter does the same for the existing OpenCode event
stream.

### 6.2 Provider config schema

```js
/**
 * @typedef {Object} ProviderConfig
 * @property {string}  id              unique identifier
 * @property {string}  displayName     shown in settings UI
 * @property {'anthropic-messages'|'openai-chat'|'opencode-runtime'} protocol
 * @property {Object<string, ModeConfig>} modes   { 'coding-plan': ..., 'api': ... }
 * @property {AuthConfig} auth         { headerName, scheme: 'bearer'|'x-api-key' }
 * @property {string}  [versionHeader] e.g. 'anthropic-version: 2026-01-01'
 * @property {ModelInfo[]} models      built-in known models
 * @property {string}  defaultModel    id of the model picked by default
 * @property {boolean} [desktopOnly]   true for opencode-legacy
 * @property {Object}  [region]        { current: 'cn'|'intl', cnUrl, intlUrl }
 * @property {ProviderQuirks} [quirks]
 *
 * @typedef {Object} ModeConfig
 * @property {string} label            UI label
 * @property {string} baseUrl          default base URL (region override may apply)
 * @property {string} [planUrl]        link to purchase/manage subscription
 * @property {string} [recommendedModel]
 *
 * @typedef {Object} AuthConfig
 * @property {string} headerName       e.g. 'Authorization', 'x-api-key'
 * @property {'bearer'|'raw'} scheme   'bearer' → "Bearer <key>", 'raw' → "<key>"
 *
 * @typedef {Object} ProviderQuirks
 * @property {string[]} [unsupportedParams]  silently dropped before request
 * @property {boolean} [streamingTolerant]   SSE parser ignores unknown events
 * @property {string}  [requestPathOverride] e.g. some endpoints use non-standard path
 */
```

### 6.4 Two-mode UX (Coding Plan vs. API)

For any provider that offers a coding plan subscription, settings show a
mode selector:

- **Coding Plan mode** — UI surfaces a "购买/管理套餐 →" link, recommends
  the tier of model that's included, optionally shows "quota remaining" if
  the provider exposes it.
- **API mode** — UI emphasizes pay-per-token pricing and recommends a
  cost-efficient model.

**The underlying HTTP call is identical in both modes.** Whether the user's
key is bound to a coding plan or a pay-per-token account is determined
server-side by the provider; FLOWnote does not need to know or care. This
keeps the code path single while giving the user a clear mental model.

Some providers (e.g. 智谱) expose a **separate base URL for coding-plan
traffic** (`/api/coding/paas/v4` vs `/api/anthropic`). When such a URL is
documented by the provider, the corresponding mode's `baseUrl` points to it.
This is configuration data, not client identity spoofing.

### 6.5 Auth header semantics

Two patterns we support, controlled by `provider.auth`:

```js
// Anthropic native
{ headerName: 'x-api-key', scheme: 'raw' }
// Most Chinese providers, OpenAI, custom OpenAI-compat
{ headerName: 'Authorization', scheme: 'bearer' }   // "Bearer <key>"
```

Plus the optional `versionHeader` (e.g. `anthropic-version: 2026-01-01` for
Anthropic native; ignored or accepted by Chinese providers).

### 6.6 Provider quirks

Real engineering hazards documented in Appendix A:

- **MiniMax**: silently drops `thinking`, `top_k`, `stop_sequences`,
  `service_tier`, `mcp_servers`, `context_management`, `container`. Our
  client strips these before the request when calling MiniMax.
- **MiniMax SSE format**: deviates slightly from Anthropic's canonical
  event union (missing `ping`, occasionally different `message_delta`
  shape). Our SSE parser runs in tolerant mode for any provider with
  `quirks.streamingTolerant = true`: unknown event types are ignored,
  missing fields default sensibly.
- **智谱 coding plan endpoint** path is non-standard
  (`/api/coding/paas/v4/chat/completions`, no `/v1` segment) when using
  the OpenAI-compat path. The Anthropic-compat path (`/api/anthropic`)
  is well-formed.

### 6.7 Provider interface contract

```js
/**
 * @typedef {Object} CreateMessageInput
 * @property {string} model
 * @property {ProviderMessage[]} messages
 * @property {string|ContentBlock[]} system
 * @property {ToolSpec[]} tools
 * @property {number} maxTokens
 * @property {number} [temperature]
 * @property {AbortSignal} [signal]
 */

/**
 * Every Provider exposes:
 *
 *   id, displayName, protocol, capabilities, models
 *
 *   createMessage(input): AsyncIterable<StreamEvent>
 *     yields normalized Anthropic-shape events
 *
 *   countTokens(messages): Promise<number>   best-effort
 *
 *   testConnection(): Promise<{ok, error?, latencyMs}>
 */
```

The agent loop only sees this interface. Adding a new provider is a new
entry in the registry (data) plus, if it uses a new protocol, a new adapter
file. Existing providers like the eight above all share two adapter files
(`anthropic-messages-adapter.js`, `openai-chat-adapter.js`) — the lineup
above adds zero new code; it's pure config.

## 7. Tool registry & contract

`runtime/agent/tool-registry.js`:

```js
const TOOL_DEFAULTS = {
  isEnabled:        () => true,
  isReadOnly:       () => false,   // fail-closed: assume side-effects
  isDestructive:    () => false,
  isConcurrencySafe:() => false,
  checkPermissions: async () => ({ behavior: 'allow' }),
  validate:         async () => ({ ok: true }),
  userFacingName:   (input) => '',
};

/**
 * @typedef {Object} ToolDef
 * @property {string} name              kebab-case identifier shown to model
 * @property {string} description       prompt shown to the model
 * @property {Object} inputSchema       JSON Schema for the tool input
 * @property {(input,ctx)=>boolean} [isEnabled]
 * @property {(input)=>boolean}     [isReadOnly]
 * @property {(input)=>boolean}     [isDestructive]
 * @property {(input)=>boolean}     [isConcurrencySafe]
 * @property {(input,ctx)=>Promise<PermissionResult>} [checkPermissions]
 * @property {(input,ctx)=>Promise<ValidationResult>} [validate]
 * @property {(input,ctx)=>AsyncIterable<ProgressOrResult>}   execute
 * @property {(input)=>string}      [userFacingName]
 */

export function buildTool(def) {
  return { ...TOOL_DEFAULTS, ...def };
}

export const toolRegistry = new Map();   // name → Tool
export function registerTool(tool) { toolRegistry.set(tool.name, tool); }
```

`execute` is an **async generator** yielding:

- `{ type: 'progress', message, data? }` — UI hint, optional
- `{ type: 'result', content, isError? }` — terminal result becoming the
  `tool_result` content block in the next API turn

Pattern lifted from Claude Code's `ToolDef`. Defaults fail-closed.

### 7.1 Tool catalog (the seven)

| Tool name | Replaces | Read-only? | Destructive? | Implementation |
|---|---|---|---|---|
| `vault_read` | `FileReadTool` | ✅ | ❌ | `app.vault.cachedRead(file)` |
| `vault_write` | `FileWriteTool` | ❌ | ✅ if exists | `vault.create` or `vault.modify` |
| `vault_edit` | `FileEditTool` | ❌ | ✅ | Read → string replace → `vault.process()` (atomic) |
| `vault_list` | `LSTool`+`GlobTool` | ✅ | ❌ | `vault.getMarkdownFiles()` + glob + frontmatter filter |
| `vault_search` | `GrepTool` | ✅ | ❌ | Pass 1: `metadataCache` (tags/links/headings). Pass 2: content scan via `cachedRead` |
| `ask_user` | `AskUserQuestionTool` | ✅ | ❌ | Inline `<select>` / `<button>` panel in chat view |
| `skill_invoke` | `SkillTool` | depends | depends | Loads `bundled-skills/<name>/SKILL.md`, runs as constrained sub-turn (§8) |

Web access (`web_fetch`) deferred to 0.6.0.

### 7.2 Permission model

Three behaviors from `checkPermissions(input, ctx)`:

- `{ behavior: 'allow' }` — proceed silently
- `{ behavior: 'deny', reason }` — return as `tool_result.is_error = true`
- `{ behavior: 'ask', summary, choices? }` — pause loop, show UI panel,
  wait for user. Choices default to `['allow once', 'allow for this
  conversation', 'deny']`. Per-conversation grants live on
  `SessionState.grants`.

Defaults:

- All `isReadOnly` tools → `allow`.
- `vault_write` to a new path → `ask` (first time per conversation).
- `vault_write` / `vault_edit` to existing path → `ask`.
- `skill_invoke` → `allow` (the skill itself can prompt).

Settings exposes global `permissionMode`:
`acceptAll` | `acceptReadOnly` (default) | `askAlways`.

## 8. Skill runner

A skill is a directory under `bundled-skills/<name>/` with:

- `SKILL.md` — the prompt
- `references/*` — long-form references the model reads on demand via
  `vault_read` after the skill prompt directs it
- `assets/*` — templates used by the workflow

`runtime/agent/skill-runner.js` orchestrates `skill_invoke`:

```
on tool_use { name: 'skill_invoke', input: { skill: 'ah-card', args } }:
  1. Resolve skill = loadBundledSkill('ah-card')
     → { promptText, referencesRoot, allowedTools, model? }
  2. Build sub-conversation:
       systemAppendix = promptText + skillContextHeader
       tools          = filterRegistry(allowedTools)
       initialUser    = `Skill arguments: ${args}`
  3. Run agent-loop with this sub-conversation
       (own turn budget, max 20 turns, same provider, tagged with skillId)
  4. Final assistant text → tool_result.content
```

Parent loop sees this as one `tool_use` → `tool_result` round-trip, same as
any other tool call. UI: collapsible "Used skill: ah-card" panel.

**No change to `bundled-skills/` source**: every `SKILL.md` you already
wrote runs unchanged. `allowedTools` parsed from the skill's frontmatter,
defaults to all vault tools if missing.

### 8.1 Skill discovery & sync

- Bundled skills ship inside the plugin (already UTF-8 embedded since
  0.4.0).
- On plugin load, sync to **`<vault>/.flownote/skills/<name>/`** (renamed
  from `.opencode/skills/` to drop the OpenCode reference).
- Skill loader prefers vault copy (user-editable) over bundled fallback.

### 8.2 Migration: `.opencode/skills/` → `.flownote/skills/`

One-time silent migration on first launch of 0.5.0:

```
if vault has .opencode/skills/ but no .flownote/skills/:
  copy .opencode/skills/* → .flownote/skills/*
  leave .opencode/skills/ in place (user may still use OpenCode mode)
  log: "Migrated skill directory to .flownote/skills/"
```

Idempotent. If both dirs exist, prefer the newer one and log a note in
diagnostics.

## 9. Agent loop

`runtime/agent/agent-loop.js`. Pseudocode (~150 lines real):

```js
async function* runAgentLoop({ provider, session, system, userMessage, tools, ctx }) {
  session.append(userMessage);

  for (let turn = 0; turn < ctx.maxTurns; turn++) {
    const stream = provider.createMessage({
      model:    session.model,
      messages: session.toApiMessages(),
      system,
      tools:    tools.map(t => t.toJsonSchemaSpec()),
      maxTokens: session.maxTokensPerTurn,
      signal:    ctx.signal,
    });

    const assistantBlocks = [];
    const toolUses        = [];

    for await (const ev of stream) {
      yield { type: 'stream', event: ev };
      if (ev.type === 'content_block_stop' && ev.content_block.type === 'tool_use') {
        toolUses.push(ev.content_block);
      }
      // ... text accumulation
    }

    session.appendAssistant(assistantBlocks);

    if (toolUses.length === 0) return;

    const toolResults = [];
    const [parallel, serial] = partition(toolUses, u => isReadOnlyToolUse(u));

    const parallelResults = await Promise.all(parallel.map(u => runTool(u, ctx)));
    for (const u of serial) toolResults.push(await runTool(u, ctx));

    session.appendUser([...parallelResults, ...toolResults]);
  }

  yield { type: 'error', error: new MaxTurnsExceeded() };
}
```

### 9.1 Streaming UX

Each stream event is forwarded to the chat view, which handles:

- `content_block_delta` (text) → append to live message bubble
- `tool_use` block start → render "🛠 Using `vault_read`…" chip
- `tool_use` block stop → show result preview

Existing `runtime/view/` rendering needs minimal change — only the event
shape adapter.

### 9.2 Cancellation & retry

- `AbortController` propagates to `requestUrl`.
- Retry on `5xx` and network errors: 3 attempts, exponential backoff
  (300ms · 900ms · 2700ms), **only** for the request to the model API.
- Tool errors are NOT retried — returned to the model as
  `tool_result.is_error`, the model decides what to do.

### 9.3 Context budget

Modern providers ship 200K–1M context. v0.5.0 does the simple thing:

- Track approximate token count per session (heuristic: 3.5 chars CJK,
  4 chars Latin per token).
- At 80% of model window, non-blocking banner: "Conversation is getting
  long — consider /compact or starting a new session."
- `/compact` is user-invocable: asks the model to summarize the
  conversation, replaces older turns with the summary, keeps the last
  4 turns verbatim.
- No auto-compact in v0.5.0.

## 10. Session storage

Anthropic-shape is canonical on disk:

```jsonc
// data/sessions/<id>.json
{
  "id": "uuid",
  "createdAt": "...",
  "providerId": "deepseek",
  "providerMode": "api",
  "model": "deepseek-v4-flash",
  "skillContext": null,
  "grants": { "vault_write:*": "session" },
  "messages": [
    { "role": "user",      "content": [{ "type": "text", "text": "..." }], "ts": ... },
    { "role": "assistant", "content": [...], "stop_reason": "end_turn", "ts": ... },
    { "role": "user",      "content": [{ "type": "tool_result", "tool_use_id": "...", "content": "..." }] }
  ]
}
```

OpenAI-compat sessions cost nothing extra: we always store the normalized
shape; the provider adapter translates only at egress.

State-migration runs on first 0.5.0 load to rewrite legacy OpenCode-shape
sessions into the new canonical shape.

## 12. Mobile considerations

| Concern | Resolution |
|---|---|
| No Node APIs (`fs`, `child_process`, `os`) | All vault tools use `app.vault.*`. No node-only deps in the new code path. |
| CORS | All upstream calls via `requestUrl` — bypasses CORS. |
| Streaming | Provider abstracts SSE; design for streaming first (1-day M1 spike on iOS confirms feasibility). If a specific provider's stream breaks on mobile, fall back per-provider; not blanket. |
| API key storage | Plugin `data.json` — same place desktop uses. |
| Skill files | UTF-8 markdown in vault. Mobile reads via Vault API. |
| File size | Bundle grows from ~1.5 MB to ~1.7–1.9 MB. Within tolerance. |

`manifest.json` stays `isDesktopOnly: false`. OpenCode provider internally
flags itself desktop-only and disables on mobile; every other provider
works on both.

## 14. Risks & mitigations

| Risk | Likelihood | Mitigation |
|---|---|---|
| `requestUrl` streaming differs desktop vs mobile | Medium | Provider abstracts. Mobile fallback returns full response after API completes (no streaming UX), still functional. M1 spike confirms. |
| Tool-call accuracy with smaller / older models | Medium | Document recommended models per provider. UI warns when a known-weak model is picked. |
| Skill sub-loop turn explosion | Low | Hard cap 20 sub-turns per skill_invoke. |
| Existing user breaks on upgrade | Low | OpenCode legacy provider stays. Auto-migration sets provider correctly. State-migration rewrites session shape. |
| Bundle size scorecard regression | Low | Budgeted +65 KB. |
| **MiniMax param drops break tool calls silently** | Medium | Strip unsupported params per `quirks.unsupportedParams` before request. Log dropped params in diagnostics. |
| **SSE format drift between providers** | Medium | `quirks.streamingTolerant` mode in SSE parser. Unknown events ignored, missing fields default sensibly. Integration test per provider against recorded fixtures. |
| Anthropic API ToS (impersonation) | None | We do not impersonate. Chinese provider coding plans use those providers' own endpoints + the user's own subscription key. |
| Provider endpoint rename / deprecation (e.g. DeepSeek retired `deepseek-chat` 2026-07-24) | Medium | Provider configs are versioned. Settings UI surfaces a "model deprecated" warning. Diagnostics check on every plugin load. |

## 15. Testing strategy

- **Unit tests** under `tests/runtime/agent/`:
  - tool-registry defaults are fail-closed
  - each tool's `validate` rejects bad input
  - each tool's `checkPermissions` returns the expected shape
  - protocol adapters: round-trip Anthropic ↔ OpenAI shape ↔ Anthropic
- **Integration tests** with `MockProvider` scripting fixed event streams —
  covers agent loop, skill-runner, permission gate.
- **Per-provider fixture tests**: for each built-in provider, record one
  real-server response (text + tool-use + multi-turn) into a fixture, replay
  through our parser to verify shape compatibility. Update fixtures when a
  provider's API changes.
- **Live smoke test** (manual, env-gated):
  one round-trip through each built-in provider that the maintainer has
  credentials for. Not on CI by default. Gated by `FLOWNOTE_E2E=1` + keys.
- **Mobile smoke**: install on iOS + Android, run `ah-note`, `ah-card`,
  `ah-review` under DeepSeek V4 Flash + (one coding-plan provider that
  works on mobile, likely 智谱 or Moonshot).

CI gates:
- `npm run ci` runs unit + mock-integration suites + fixture replay.
- `npm run test:e2e` (opt-in): live smoke if env set.
