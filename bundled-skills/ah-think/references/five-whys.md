# 5个为什么 (5 Whys)

> "没有问题是小问题，每个问题背后都有根本原因。找到根本原因，才能真正解决问题。"
> — 大野耐一（丰田生产方式之父）

---

## 背景

"5个为什么"（5 Whys）是丰田汽车公司创始人丰田佐吉（Sakichi Toyoda）发明的问题分析技术。它是丰田生产系统（TPS）的核心工具之一。

大野耐一（Taiichi Ohno），丰田生产方式的设计师，称其为"丰田科学方法的基础"。

**核心洞见**：我们常常只治标不治本，因为我们停留在问题的表面症状，没有挖到根本原因。连续问"为什么"，能帮你一层层深入，直到找到真正的问题。

---

## 核心原理

### 表面症状 vs 根本原因

很多问题解决之所以无效，是因为只处理了**表面症状**而非**根本原因**。

**例子**：

| 表面症状 | 一般对策 | 根本原因 | 真正对策 |
|---------|---------|---------|---------|
| 总是迟到 | 设更多闹钟 | 晚睡导致起不来 | 调整睡眠时间 |
| 写作写不出来 | 强迫自己写 | 害怕被评判 | 处理心理障碍 |
| 总是忘记事情 | 记更多笔记 | 同时处理太多事 | 简化任务量 |

### 为什么问5次？

5不是固定的数字。关键是**一直问到找到可以行动的根本原因**。有时3次就够，有时需要7-8次。

"5"是一个经验数字：通常5次追问能挖到足够深。

---

## 操作方法

### 基本流程

1. **陈述问题**：清晰描述你观察到的问题/现象
2. **问"为什么"**：为什么会发生这个问题？
3. **继续追问**：对答案再问"为什么"
4. **重复**：直到找到根本原因（通常5次左右）
5. **验证**：这个根本原因解决了，问题会消失吗？

### 完整示例

**问题**：项目延期了

```
为什么1：为什么项目延期了？
→ 因为测试阶段花的时间比预期长

为什么2：为什么测试阶段花的时间比预期长？
→ 因为发现了很多bug需要修复

为什么3：为什么会有这么多bug？
→ 因为开发阶段时间太赶，没有做充分的代码审查

为什么4：为什么开发阶段时间太赶？
→ 因为项目开始时需求不清楚，后来变更了很多次

为什么5：为什么需求开始时不清楚？
→ 因为项目启动前没有和客户充分沟通确认

根本原因：项目启动流程缺乏充分的需求确认环节
对策：改进项目启动流程，增加需求确认检查点
```

## AI引导模板

### 通用模板

```
你观察到的问题是：「XX」

让我们一起找到根本原因：

❓ 为什么1：为什么会出现XX？
[等待用户回答]

❓ 为什么2：为什么[用户的回答]？
[等待用户回答]

❓ 为什么3：为什么[用户的回答]？
[等待用户回答]

... 继续追问，直到：
- 用户回答触及了可以行动的根本原因
- 或者继续追问也不会有新发现

检验：如果解决了这个根本原因，原来的问题会消失吗？
```

### 引导技巧

**追问方式**：
- "为什么会这样？"
- "这是什么导致的？"
- "再往前追溯，原因是什么？"

**验证方式**：
- "如果这个原因解决了，问题会消失吗？"
- "这个原因是你能改变的吗？"
- "这是最根本的原因，还是还可以再往前追？"

## 阅读笔记中的应用

### 场景1：理解一个问题的根源

```
书中讨论了「XX问题」。

🔍 5 Whys 深挖：

作者说XX是个问题。让我们理解为什么：

- 为什么XX是个问题？它导致了什么？
- 为什么会有XX？它的原因是什么？
- 为什么那个原因存在？更深层是什么？

理解根本原因，比记住表面现象更有价值。
```

### 场景2：分析自己的困惑

```
读完后，你感到困惑/不舒服/不认同。

🔍 5 Whys 自我探索：

为什么这段内容让你[困惑/不舒服/不认同]？

- 为什么1：是因为和你原有认知冲突？
- 为什么2：为什么你会有那个原有认知？
- 为什么3：那个认知来自哪里？
- ...

找到根源，可能会有新发现。
```

### 场景3：把洞见变成行动

```
你的洞见是「应该XX」，但一直没做到。

🔍 5 Whys 找障碍：

为什么你知道应该XX但没有做？

- 为什么1：是没时间？没动力？不知道怎么做？
- 为什么2：为什么会那样？
- 为什么3：再往深挖...

找到真正的障碍，才能真正改变。
```

## 引导问题清单

**基础追问**：
- "为什么？"
- "这是什么导致的？"
- "更深层的原因是什么？"
- "还能再往前追吗？"

**验证根因**：
- "这是你能改变的吗？"
- "如果这个解决了，原问题会消失吗？"
- "这是最根本的原因了吗？"

**防止偏离**：
- "这个回答是在解释原因，还是在描述另一个症状？"
- "我们是不是岔开了？回到主线"

## 注意事项

### ✅ 做到

- 每次"为什么"都基于前一个回答
- 追问到可以行动的层面
- 验证找到的是否真是根因
- 保持好奇心，不带评判

### ❌ 避免

**常见错误1：停得太早**
```
问题：为什么报告迟交了？
回答：因为我忘了。
（停止）← 这不是根因！应该继续问"为什么会忘？"
```

**常见错误2：分叉太多**
```
问题：为什么项目失败？
回答1：因为预算不够
回答2：因为团队人手不足
回答3：因为需求不清楚
（同时追三条线）← 选一条最关键的深入
```

**常见错误3：停在无法改变的原因**
```
为什么迟到？→ 因为堵车 → 因为车太多 → 因为城市规划问题
（这你改变不了）← 应该追问"为什么没预留堵车时间"
```

**常见错误4：变成指责**
```
为什么出错？→ 因为他不认真
（这是评判，不是原因）← 应该问"是什么导致他出错"
```

---

## 与其他模型的配合

- **5 Whys + 逆向思维**：问"为什么会失败"然后用5 Whys深挖
- **5 Whys + 第一性原理**：挖到根因后，用第一性原理重新构建
- **5 Whys + 二阶思考**：找到根因后，思考解决根因的二阶效应

---

## 退出条件

当以下情况出现时，可以结束引导：

- 用户找到了可以行动的根本原因
- 继续追问已经不会有新发现
- 用户表示"这就是根因""我明白了"

AI应说：
```
很好，我们找到了问题的根本原因：「XX」。

这是你可以采取行动的层面。要记录这个分析过程吗？
```

## 经典案例

### 丰田的机器停机案例

**问题**：机器停机了

```
为什么1：为什么机器停了？
→ 因为保险丝烧了

为什么2：为什么保险丝烧了？
→ 因为电流过载

为什么3：为什么电流过载？
→ 因为轴承缺乏润滑

为什么4：为什么轴承缺乏润滑？
→ 因为润滑油泵没有打够油

为什么5：为什么润滑油泵没有打够油？
→ 因为油泵进油口被金属碎屑堵塞

根本原因：进油口没有过滤装置
对策：给进油口加装过滤器
```

如果只解决"保险丝烧了"，问题会反复出现。找到根因，才能彻底解决。
