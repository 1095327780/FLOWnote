# 项目归档引导技能

## 概述

本技能帮助用户在项目完成后进行有意义的复盘，提炼有价值的知识，并将项目规范地归档。

**核心价值**：
- 引导用户做项目复盘，不浪费经验
- 识别值得转化为永久笔记的内容
- 自动移动项目到正确的归档位置
- 更新领域页的项目状态

## 归档原则

> **归档不是删除，而是让项目"退休"**

- 项目归档后依然可以搜索和引用
- 归档前应该提炼出有价值的知识
- 归档按照所属领域分类存放

## 工作流程

```
1. 确认要归档的项目（AI询问）
   ↓
2. 检查项目完成度（AI检查）
   ↓
3. 引导项目复盘（AI引导）
   ↓
4. 识别可提炼内容（AI分析）
   ↓
5. 更新项目状态（AI自动）
   ↓
6. 移动到归档目录（AI自动）
   ↓
7. 更新领域页（AI自动）
   ↓
8. 汇报结果（AI汇报）
```

## 阶段一：确认项目

### 1.1 询问项目

```
你想归档哪个项目？

当前活跃项目：
| 编号 | 名称 | 状态 |
|------|------|------|
| 26-01 | 内容更新规划 | 进行中 |
| 26-02 | B站视频制作 | 进行中 |
| ...  | ... | ... |

请输入项目编号或名称。
```

### 1.2 确认项目信息

```
确认要归档的项目：

📁 **项目**：{{编号}} {{项目名称}}
🌱 **所属领域**：{{领域}}
📅 **开始日期**：{{日期}}
🎯 **原定目标**：{{目标}}

确认是这个项目吗？
```

## 阶段二：检查完成度

### 2.1 检查成功标准

AI 读取项目主页，检查成功标准的完成情况：

```
让我检查一下项目的完成度...

**成功标准检查**：
- [x] 标准1 ✅
- [ ] 标准2 ❌ 未完成
- [x] 标准3 ✅

完成率：2/3 (67%)

{{如果有未完成项}}
有 {{N}} 项成功标准未完成。你确定要归档吗？

1. 继续归档（标记为「部分完成」）
2. 暂不归档，先完成剩余标准
```

---

## 阶段三：项目复盘

### 3.1 复盘问题

```
在归档之前，让我们做一个简单的项目复盘。

**问题1：这个项目最终达成了什么成果？**
用一句话概括。

**问题2：项目过程中最大的收获是什么？**
可以是技能、认知、经验等。

**问题3：如果重新做这个项目，你会做什么不同？**
（不是自责，而是提炼经验）

**问题4：这个项目中有哪些内容值得记录下来？**
- 方法论
- 工具技巧
- 认知突破
- 踩过的坑
```

### 3.2 AI 总结

根据用户回答，AI 生成项目总结：

```
📋 **项目总结**

**最终成果**：{{用户回答}}

**关键收获**：
- {{收获1}}
- {{收获2}}

**经验教训**：
- {{教训1}}

**可提炼内容**：
- {{内容1}} — 可能值得转为永久笔记
- {{内容2}} — 工具技巧，可以归档保留

这个总结准确吗？需要补充什么？
```

## 阶段四：识别可提炼内容

### 4.1 分析项目内容

AI 扫描项目文件夹，识别有价值的内容：

```
让我扫描一下项目中的文件...

**项目文件**：
- 📍 项目总览.md
- 需求文档.md
- 技术方案.md
- 会议记录/

**发现可能有价值的内容**：

💡 **可能值得制卡**（永久笔记）：
1. 「{{内容}}」— 来自技术方案.md
2. 「{{洞察}}」— 来自你刚才的复盘

📦 **建议保留**（归档后可搜索）：
- 需求文档 — 未来类似项目可参考
- 技术方案 — 技术实现细节

你想现在处理这些内容吗？还是先归档，以后再处理？
```

### 4.2 处理选项

```
选择处理方式：

1. 🚀 **现在处理** — 用 `/ah-card` 把有价值的内容转为永久笔记
2. 📦 **先归档** — 在项目主页标记「待提炼」，以后再处理
3. ⏭️ **跳过** — 直接归档，不做额外处理
```

## 阶段五：更新项目状态

### 5.1 更新项目主页

在项目主页的「完成归档」区块填写信息：

```markdown
## 📦 完成归档

**完成日期**：{{YYYY-MM-DD}}
**最终成果**：{{用户描述的成果}}
**经验总结**：
- {{经验1}}
- {{经验2}}

**待提炼内容**（如有）：
- [ ] {{内容1}} — 转为永久笔记

**归档位置**：[[04-创造层/Archives/{{领域}}/{{编号}} {{项目名称}}]]
```

### 5.2 更新 YAML

```yaml
---
状态: 已完成
完成日期: {{YYYY-MM-DD}}
---
```

## 阶段六：移动到归档目录

### 6.1 归档位置规则

根据项目所属领域，归档到对应分类：

```
04-创造层/Archives/
├── 内容创作/        ← 🌱 内容创作 的项目
├── 产品开发/        ← 🌱 产品开发 的项目
├── 个人成长/        ← 🌱 个人成长 的项目
├── 效率工具/        ← 🌱 效率工具 的项目
└── 其他/            ← 不属于以上领域的项目
```

### 6.2 多领域项目

如果项目属于多个领域，归档到**主要领域**：

```
项目「26-05 SnapPlan开发」属于多个领域：
- 🌱 产品开发
- 🌱 效率工具

请选择主要领域作为归档位置：
1. 产品开发（推荐 — 因为是软件产品）
2. 效率工具
```

### 6.3 执行移动

```bash
mv "04-创造层/Projects/{{编号}} {{项目名称}}" "04-创造层/Archives/{{领域}}/{{编号}} {{项目名称}}"
```

## 阶段七：更新领域页

### 7.1 移动项目链接

在领域页中，将项目从「活跃项目」移到「历史归档」：

```markdown
## 🚀 关联项目

### 活跃项目
| 项目 | 产出形式 | 核心知识 |
|------|----------|----------|
| ... | ... | ... |

### 历史归档
| 项目 | 完成时间 | 主要成果 |
|------|----------|----------|
| [[Archives/{{领域}}/{{编号}} {{项目名称}}/📍 项目总览\|{{项目名称}}]] | {{日期}} | {{成果}} |
```

## 注意事项

1. **不要强制复盘**：如果用户没时间，可以快速归档
2. **保留原始文件**：归档是移动，不是删除
3. **标记待提炼**：有价值的内容即使没时间处理也要标记
4. **双向更新**：同时更新项目主页和领域页
5. **路径正确**：确保归档路径存在
